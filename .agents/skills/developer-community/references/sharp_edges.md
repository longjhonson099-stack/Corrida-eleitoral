# Developer Community - Sharp Edges

## Docs Drift

### **Id**
docs-drift
### **Summary**
Documentation becomes outdated and misleading
### **Severity**
high
### **Situation**
Docs don't match current API/product
### **Why**
  Developers get frustrated.
  Support tickets increase.
  Trust erodes.
  "If docs are wrong, what else is?"
  
### **Solution**
  ## Preventing Docs Drift
  
  ### Automation
  - Auto-generate API reference from code
  - CI checks for doc links
  - Version docs with releases
  - Automated screenshot capture
  
  ### Process
  - Docs review in PR checklist
  - Quarterly docs audit
  - "Edit this page" links
  - Community can submit fixes
  
  ### Ownership
  - Every feature has doc owner
  - Docs are part of "done"
  - Track doc freshness metrics
  - Celebrate doc contributions
  
### **Symptoms**
  - Support tickets about documented features
  - "Docs are outdated" complaints
  - Tutorials that don't work
  - Code samples that fail
### **Detection Pattern**
outdated|doesn't work|docs wrong|broken example

## Contributor Burnout

### **Id**
contributor-burnout
### **Summary**
Active contributors stop contributing
### **Severity**
high
### **Situation**
Key community members disengage
### **Why**
  Unresponsive maintainers.
  Contributions ignored.
  Toxic interactions.
  Burnout from over-commitment.
  
### **Solution**
  ## Preventing Contributor Burnout
  
  ### Recognition
  - Thank every contribution
  - Public recognition
  - Contributor swag
  - Conference sponsorships
  
  ### Sustainability
  - Don't expect free work forever
  - Offer paid opportunities
  - Clear commitment expectations
  - Sabbatical acceptance
  
  ### Support
  - Mentor new contributors
  - Share maintainer load
  - Async-friendly culture
  - Mental health awareness
  
  ### Warning Signs
  - Response times increasing
  - Shorter responses
  - Skipping meetings
  - Negative tone creeping in
  
### **Symptoms**
  - PRs sitting unreviewed
  - Active members going quiet
  - Complaints about workload
  - Maintainers leaving
### **Detection Pattern**
burned out|too much|stepping back|taking a break

## Enterprise Vs Community

### **Id**
enterprise-vs-community
### **Summary**
Enterprise customers overshadow community
### **Severity**
medium
### **Situation**
Paying customers get all attention, community ignored
### **Why**
  Revenue pressure.
  Enterprise support contracts.
  Community seen as "free" tier.
  
### **Solution**
  ## Balancing Enterprise and Community
  
  ### Dedicated Resources
  - Separate DevRel for community
  - Community roadmap input
  - Community-specific features
  - Don't deprioritize community bugs
  
  ### Transparency
  - Explain enterprise-only features
  - Community knows the deal
  - Show community value to business
  - Celebrate community wins publicly
  
  ### Give Back
  - Enterprise funds OSS work
  - Community contributions benefit all
  - Open source non-differentiating code
  - Community advisory board
  
### **Symptoms**
  - Only enterprise gets fixes
  - Community features delayed
  - Less community engagement
  - Fork threats
### **Detection Pattern**
enterprise only|community ignored|paid features|fork

## Devrel Without Influence

### **Id**
devrel-without-influence
### **Summary**
DevRel team can't influence product
### **Severity**
high
### **Situation**
Community feedback doesn't reach product team
### **Why**
  DevRel siloed from product.
  Feedback not systematized.
  Product team doesn't value community input.
  
### **Solution**
  ## DevRel Influence
  
  ### Organizational
  - DevRel reports to or with Product
  - Regular product sync
  - DevRel in sprint planning
  - Shared OKRs with product
  
  ### Feedback Loop
  - Structured feedback submission
  - Monthly community insights report
  - Quantify developer pain points
  - Close the loop publicly
  
  ### Credibility Building
  - DevRel ships features too
  - Technical credibility
  - Data-driven arguments
  - Win small battles first
  
### **Symptoms**
  - Same issues reported repeatedly
  - Community feels unheard
  - DevRel frustration
  - Gap between promises and delivery
### **Detection Pattern**
not listening|ignored feedback|still broken|nothing changes
